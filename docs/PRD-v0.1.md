# 局域网文件传输工具 PRD v0.1

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 局域网文件传输工具 |
| 版本 | v0.1 |
| 文档类型 | 初版 PRD |

| 技术方向 | Tauri 2 + React + TypeScript + Tailwind CSS + Rust(axum/tokio) + HTTP 文件传输 |

## 2. 背景与问题

用户拥有多台电脑设备，日常存在局域网内互传文件的需求。当前常见做法是：

1. 在 A 电脑上将文件发送到微信文件助手。
2. 在 B 电脑上登录微信。
3. 在 B 电脑上重新下载文件。

该流程依赖第三方应用和外部服务，步骤较多，且不符合局域网内设备之间直接传输文件的直觉。

本项目希望开发一个轻量的桌面端小工具，使同一局域网内的设备可以直接传输文件，减少中转、登录、下载等不必要步骤。

## 3. 产品定位

本产品是一个面向个人多设备场景的局域网文件传输工具。

v0.1 版本聚焦于最小可用闭环：

> 在同一局域网内，两台桌面设备分别启动应用后，发送方可以通过手动输入接收方地址，将一个或多个文件直接发送给接收方；接收方确认后保存文件，并展示传输状态。

## 4. 目标用户

### 4.1 核心用户

- 拥有多台电脑设备的个人用户。
- 经常需要在局域网内传输文件的开发者、设计师、办公用户。
- 希望避免通过微信、网盘、聊天工具中转文件的用户。

### 4.2 当前优先用户画像

当前产品首先服务于开发者本人：

- 熟悉 Web 前端技术栈。
- 能接受手动输入 IP 地址、端口等轻量配置。
- 更关注工具是否可用、轻量、直接，而不是一开始就具备完整商业产品体验。

## 5. 产品目标

### 5.1 v0.1 目标

v0.1 的核心目标是跑通局域网文件传输的最小闭环。

具体目标：

- 同一局域网内两台设备可以建立传输关系。
- 发送方可以选择或拖拽文件进行发送。
- 接收方可以确认或拒绝接收。
- 文件可以直接保存到接收方本地目录。
- 双方可以看到基础传输状态。
- 整体流程比“微信文件助手中转”更短、更直接。

### 5.2 非目标

v0.1 暂不追求：

- 公网传输。
- 用户账号体系。
- 云端中转。
- 移动端 App。
- 断点续传。
- 文件夹传输。
- 自动局域网设备发现。
- 完整端到端加密。
- 多人群组传输。
- 剪贴板同步。

## 7. 核心使用流程

### 7.1 发送方流程

1. 用户打开应用。
2. 应用展示本机设备名和本机局域网地址。
3. 用户输入接收方地址。
4. 用户选择或拖拽一个或多个文件。
5. 用户点击发送。
6. 应用向接收方发起文件发送请求。
7. 等待接收方确认。
8. 接收方确认后开始传输。
9. 发送方查看传输进度。
10. 传输完成后展示成功状态。

### 7.2 接收方流程

1. 用户打开应用。
2. 应用启动本地 HTTP 接收服务。
3. 应用展示本机局域网访问地址。
4. 收到来自发送方的文件发送请求。
5. 用户查看发送方信息、文件名、文件大小。
6. 用户选择接收或拒绝。
7. 若接收，应用保存文件到默认目录。
8. 接收方查看接收进度。
9. 传输完成后展示成功状态。
10. 用户可打开保存目录。

## 9. 功能需求

### 9.1 本机信息展示

优先级：P0

需求说明：

- 应用启动后展示当前设备名。
- 应用启动后展示本机局域网访问地址。
- 应用启动后展示本地接收服务状态。
- 用户可以复制本机访问地址，用于在其他设备上手动连接。

#### 9.1.1 设备名称获取规则

设备名称用于帮助用户识别当前设备，主要面向人类可读展示，不作为设备唯一标识。

设备名称展示优先级：

```text
用户自定义设备名 > 系统设备名 > 默认兜底名称
```

具体规则：

- 首次启动时，应用从操作系统读取当前设备名称。
- 如果用户未设置过自定义设备名，则展示系统设备名。
- 如果用户设置过自定义设备名，则优先展示用户自定义设备名。
- 用户可以在应用内编辑自定义设备名。
- 自定义设备名保存在本地配置中。
- 当系统设备名读取失败时，使用默认兜底名称，例如 `My Device`。

示例：

| 系统设备名 | 用户自定义设备名 | 最终展示 |
| --- | --- | --- |
| `DESKTOP-8F3K2Q` | 未设置 | `DESKTOP-8F3K2Q` |
| `DESKTOP-8F3K2Q` | `ThinkBook` | `ThinkBook` |
| 读取失败 | 未设置 | `My Device` |

后续如果需要实现信任设备、设备配对等能力，应额外生成稳定的设备 ID，不应将设备名称作为唯一身份标识。

#### 9.1.2 多网卡场景下的 IP 地址选择规则

应用启动后需要枚举本机网络接口，并自动推荐一个最可能用于局域网访问的 IPv4 地址。

多网卡场景包括但不限于：

- Wi-Fi 网卡。
- 有线网卡。
- VPN 网卡。
- Docker/WSL 虚拟网卡。
- VirtualBox/VMware/Hyper-V 虚拟机网卡。
- 蓝牙或其他系统虚拟网卡。
- Loopback 地址。

这些网卡可能导致应用展示不可访问或不适合局域网传输的 IP 地址。例如，若应用误展示 Docker、WSL、VPN 或虚拟机网卡地址，另一台局域网设备可能无法连接。

v0.1 采用“自动推荐 + 候选列表 + 用户手动切换/复制”的方案。

IP 筛选规则：

- 仅选择 IPv4 地址。
- 排除 `127.0.0.1` 等 loopback 地址。
- 排除 `169.254.x.x` 等 link-local 地址。
- 排除未启用、无有效 IPv4 地址的网络接口。
- 优先选择私有 IPv4 地址：
  - `192.168.x.x`
  - `10.x.x.x`
  - `172.16.x.x` 至 `172.31.x.x`
- 优先选择 Wi-Fi、有线网卡对应的地址。
- 降低 VPN、Docker、WSL、VirtualBox、VMware、Hyper-V、vEthernet 等虚拟网卡地址的推荐优先级。

界面展示规则：

- 顶部状态区展示一个推荐访问地址。
- 如果存在多个候选 IP，提供候选地址列表。
- 用户可以复制推荐地址。
- 用户可以从候选地址中手动选择一个作为展示地址。

展示示例：

```text
推荐地址：http://192.168.1.23:7788

其他可用地址：
- 192.168.1.23  Wi-Fi
- 192.168.31.5  Ethernet
- 10.8.0.14     VPN
- 172.17.0.1    Docker
```

#### 9.1.3 本地服务监听规则

本地 HTTP 接收服务监听地址：

```text
0.0.0.0:实际端口
```

说明：

- 服务监听 `0.0.0.0`，表示接收来自本机所有网络接口的连接。
- UI 不展示 `0.0.0.0`，而是展示推荐的局域网 IP + 实际端口。
- 如果用户从候选 IP 中手动切换展示地址，只影响 UI 展示和复制地址，不影响服务监听范围。

示例：

```text
服务实际监听：0.0.0.0:7788
界面展示地址：http://192.168.1.23:7788
```

#### 9.1.4 服务端口冲突处理规则

默认端口：

```text
7788
```

端口冲突处理：

- 应用启动本地 HTTP 服务时，优先尝试监听 `7788`。
- 如果 `7788` 被占用，则自动尝试 `7789`。
- 如果仍被占用，则继续按端口号递增尝试。
- 端口尝试范围暂定为 `7788-7888`。
- 应用最终展示实际启用的端口。
- 如果尝试范围内没有可用端口，则本地服务启动失败，并展示错误提示。

示例：

```text
7788 被占用
7789 被占用
7790 可用

最终服务地址：http://192.168.1.23:7790
```

错误提示：

```text
本地接收服务启动失败，请检查端口占用或防火墙设置。
```

#### 9.1.5 后续待细化

- 设备名称的最大长度、允许字符和空值校验。
- 多 IP 候选列表的排序细则。
- 是否记住用户上一次手动选择的展示 IP。
- Windows 防火墙拦截时的提示与引导。

### 9.2 手动连接目标设备

优先级：P0

需求说明：

- 用户可以输入接收方地址。
- 用户可以输入 IP、IP + 端口或完整 URL。
- 应用自动补全协议前缀和默认端口。
- 应用应校验地址格式。
- 用户可以测试目标设备是否可连接。
- 连接测试成功后，应用展示目标设备信息。
- 连接测试成功后，应用保存最近连接设备。
- 连接失败时需要给出明确反馈。

#### 9.2.1 地址输入与自动补全规则

v0.1 仅支持 HTTP 协议，不支持 HTTPS。

用户输入地址时，应用需要兼容以下格式：

| 用户输入 | 规范化后 |
| --- | --- |
| `192.168.1.23` | `http://192.168.1.23:7788` |
| `192.168.1.23:7790` | `http://192.168.1.23:7790` |
| `http://192.168.1.23` | `http://192.168.1.23:7788` |
| `http://192.168.1.23:7790` | `http://192.168.1.23:7790` |

规则说明：

- 当用户未输入协议时，自动补全为 `http://`。
- 当用户未输入端口时，自动补全默认端口 `7788`。
- 当用户输入完整地址时，保留用户输入的端口。
- v0.1 暂不支持域名解析场景，优先支持 IPv4 地址。
- 规范化后的地址用于连接测试、文件发送和最近连接记录。

#### 9.2.2 连接测试接口

用户输入目标地址后，可以点击“测试连接”。

应用请求目标设备接口：

```http
GET /api/device
```

该接口用于确认：

- 目标地址是否可访问。
- 目标服务是否为本应用的接收服务。
- 目标设备名称是什么。
- 目标设备当前是否允许接收文件。
- 双方通信协议是否兼容。

响应示例：

```json
{
  "ok": true,
  "deviceId": "8b2fd28d-3a6a-4a86-8e3a-90ed6a01f88c",
  "deviceName": "ThinkBook",
  "version": "0.1.0",
  "protocolVersion": "0.1",
  "receiveEnabled": true
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `ok` | boolean | 是 | 接口是否正常返回 |
| `deviceId` | string | 是 | 设备稳定标识，用于识别最近连接设备 |
| `deviceName` | string | 是 | 目标设备展示名称 |
| `version` | string | 是 | 目标应用版本号，例如 `0.1.0` |
| `protocolVersion` | string | 是 | 文件传输协议版本，例如 `0.1` |
| `receiveEnabled` | boolean | 是 | 目标设备当前是否允许接收文件 |

#### 9.2.3 字段生成与判断规则

`version`：

- 表示目标设备当前运行的应用版本。
- 用于展示、调试和后续升级提示。
- 示例：`0.1.0`。

`protocolVersion`：

- 表示目标设备支持的文件传输协议版本。
- 用于判断接口、请求格式和传输流程是否兼容。
- v0.1 暂定协议版本为 `0.1`。
- v0.1 中，双方 `protocolVersion` 相同则认为兼容；不同则提示可能不兼容。

`deviceId`：

- 应用首次启动时生成 UUID v4。
- 生成后保存到本地配置。
- 后续启动继续使用同一个 `deviceId`。
- 用户修改设备名称不会影响 `deviceId`。
- 如果用户删除本地配置，`deviceId` 可以重新生成。
- 不使用设备名称、IP 地址或 MAC 地址作为 `deviceId`。

`receiveEnabled`：

- 表示目标设备当前是否允许接收文件。
- v0.1 暂不提供“暂停接收”开关。
- v0.1 的判断规则为：

```text
receiveEnabled = 保存目录可用
```

v0.1 暂不返回 `receiveDisabledReason` 字段。

#### 9.2.4 连接测试结果

连接测试成功条件：

- 请求目标设备 `/api/device` 成功。
- 响应数据结构符合本应用协议。
- `protocolVersion` 与当前应用兼容。
- `receiveEnabled` 为 `true`。

连接测试成功后，界面展示：

```text
已连接：ThinkBook
地址：http://192.168.1.23:7788
```

连接测试失败时，至少覆盖以下提示：

| 场景 | 提示 |
| --- | --- |
| 地址格式错误 | 地址格式不正确 |
| 请求超时 | 连接超时，请确认对方设备已启动应用并处于同一局域网 |
| 连接被拒绝 | 无法连接到该地址，请检查端口或防火墙 |
| 返回内容不匹配 | 目标地址不是有效的局域网传输工具 |
| 协议版本不兼容 | 对方应用版本暂不兼容 |
| 对方不可接收 | 对方设备当前不可接收文件 |

发送文件前必须完成一次成功的连接测试。

#### 9.2.5 最近连接设备

连接测试成功后，应用将目标设备保存到最近连接列表。

最近连接设备字段：

```ts
type RecentDevice = {
  deviceId?: string;
  deviceName: string;
  address: string;
  ip: string;
  port: number;
  lastConnectedAt: number;
  lastSuccessAt?: number;
};
```

保存规则：

- 连接测试成功后保存或更新最近连接记录。
- 若存在相同 `deviceId` 的记录，则更新原记录。
- 若不存在 `deviceId`，但存在相同 `address` 的记录，则更新原记录。
- 最近连接列表按最近成功连接时间倒序排列。
- 最多保存 10 条记录。
- 最近连接记录保存在本地配置中。
- 用户可以删除某条最近连接记录。

使用规则：

- 用户可以点击最近连接设备，快速填入目标地址。
- 点击最近连接设备后，仍需重新执行连接测试。
- 最近连接设备中的 IP 地址可能因网络变化失效，连接失败时应给出提示。

#### 9.2.6 后续待细化

- 最近连接设备在 UI 中的展示样式。
- 最近连接设备是否支持重命名或备注。
- 是否支持清空全部最近连接记录。
- 协议版本兼容矩阵。

### 9.3 文件选择与待发送列表

优先级：P0

需求说明：

- 用户可以通过文件选择器选择文件。
- 用户可以通过拖拽添加文件。
- 支持一次选择多个文件。
- 展示待发送文件名、大小、数量。
- 展示待发送文件总大小。
- 用户可以从待发送列表中移除文件。

#### 9.3.1 文件大小限制

v0.1 不设置单文件大小限制，也不设置总文件大小限制。

规则说明：

- 用户可以选择任意大小的单个文件。
- 用户可以一次选择多个文件。
- 应用不因文件大小主动阻止用户发送。
- 大文件传输时间取决于局域网速度、磁盘读写速度和设备性能。
- 技术实现上应避免一次性将整个文件读入内存，后续实现阶段优先采用流式上传/接收。

待发送列表需要展示：

- 文件名。
- 文件大小。
- 文件数量。
- 所有待发送文件的总大小。

展示示例：

```text
已选择 5 个文件，共 3.2 GB
```

#### 9.3.2 待发送列表规则

用户选择或拖拽文件后，文件进入待发送列表。

规则说明：

- 同一次任务中允许包含多个文件。
- 用户可以在发送前从待发送列表中移除某个文件。
- 如果待发送列表为空，则发送按钮不可用。
- 如果某个文件在发送前已被删除或不可读取，发送时应提示文件不可用。

#### 9.3.3 重名文件处理规则

重名文件由接收端用户决定是否覆盖。

处理时机：

- 发送方提交文件传输请求后，接收方在确认接收前检查目标保存目录。
- 如果目标保存目录中存在同名文件，接收方需要展示重名确认。

单文件重名时：

```text
文件已存在

接收目录中已存在同名文件：
demo.zip

是否覆盖？

[覆盖] [取消]
```

多文件重名时：

- 如果存在一个或多个重名文件，接收方统一展示重名文件列表。
- v0.1 仅支持两个操作：
  - 覆盖全部重名文件。
  - 取消本次传输任务。

处理结果：

- 用户选择覆盖：继续接收本次任务，重名文件会被覆盖。
- 用户选择取消：取消本次传输任务。
- v0.1 不支持逐个处理重名文件。
- v0.1 不支持自动重命名。

后续版本可考虑支持：

- 跳过重名文件，继续接收其他文件。
- 对每个重名文件逐个确认。
- 自动重命名，例如 `demo (1).zip`。

### 9.4 文件发送

优先级：P0

需求说明：

- 使用 HTTP 方式发送文件。
- 发送前需要先向接收方提交发送请求。
- 接收方确认后开始传输。
- 多文件按顺序逐个上传。
- 发送过程中展示进度。
- 发送完成后展示成功状态。
- 发送失败后展示失败原因。
- v0.1 不支持发送中取消。

#### 9.4.1 发送请求与实际上传

v0.1 将“发送请求”和“实际上传”拆分为两个接口。

推荐流程：

```text
1. 发送方调用 POST /api/transfer/request，提交传输请求。
2. 接收方展示文件信息、重名检查结果和接收确认。
3. 接收方确认接收后，发送方开始上传文件。
4. 发送方调用 POST /api/transfer/upload，上传文件内容。
```

拆分原因：

- 接收方可以先查看文件信息，再决定是否接收。
- 接收方可以在上传前检查重名文件。
- 如果接收方拒绝，则无需上传文件内容。
- 传输状态更清晰，便于后续扩展。

#### 9.4.2 多文件上传规则

v0.1 中，多文件属于同一个传输任务，但文件内容按顺序逐个上传。

规则说明：

- 多文件任务共享同一个传输任务 ID。
- 发送方按照待发送列表顺序逐个上传文件。
- 当前文件上传完成后，再上传下一个文件。
- v0.1 不做多文件并发上传。
- v0.1 不做批量 multipart 一次性上传所有文件。

示例：

```text
任务 A：
1. 上传 file-1.zip
2. 上传 file-2.png
3. 上传 file-3.mp4
```

#### 9.4.3 发送取消规则

v0.1 不支持发送中取消。

规则说明：

- 发送方发起传输请求后，接收方可以接收或拒绝。
- 接收方确认后，发送方开始上传。
- 上传过程中，界面不提供“取消发送”操作。
- 如果传输过程中应用关闭、网络中断或服务异常，则任务进入失败状态。

后续版本可考虑支持发送中取消、临时文件清理和部分文件恢复。

### 9.5 文件接收确认

优先级：P0

需求说明：

- 接收方收到请求后展示确认信息。
- 确认信息至少包含发送方、文件名、文件数量、文件总大小。
- 用户可以选择接收或拒绝。
- 拒绝后发送方应看到被拒绝状态。
- 接收方同一时间只处理一个传输请求。
- v0.1 不支持部分接收。

#### 9.5.1 单任务接收规则

v0.1 中，接收方同一时间只处理一个传输请求。

规则说明：

- 如果接收方当前没有待确认或传输中的任务，则可以接收新的传输请求。
- 如果接收方当前已有待确认任务，则新的传输请求直接返回忙碌状态。
- 如果接收方当前已有传输中任务，则新的传输请求直接返回忙碌状态。

忙碌状态响应示例：

```json
{
  "ok": false,
  "errorCode": "receiver_busy",
  "message": "接收方当前正在处理其他传输任务"
}
```

发送方收到忙碌状态后，任务显示为失败，并提示：

```text
对方正在处理其他传输任务，请稍后再试。
```

#### 9.5.2 接收确认规则

接收方以“传输任务”为单位进行确认。

规则说明：

- 用户可以选择接收全部文件。
- 用户可以选择拒绝整个传输任务。
- v0.1 不支持部分接收。
- v0.1 不支持只接收某几个文件。
- v0.1 不设置自动拒绝超时。
- 接收请求会等待用户手动接收或拒绝。

如果存在重名文件，按 9.3.3 的重名文件处理规则执行：

- 覆盖全部重名文件。
- 取消本次传输任务。

### 9.6 文件保存

优先级：P0

需求说明：

- 接收的文件保存到当前保存目录。
- 默认保存目录为系统下载目录下的 `LanTransfer` 文件夹。
- 用户可以在应用内修改默认保存目录。
- 保存完成后支持打开保存目录。

#### 9.6.1 保存目录规则

保存目录优先级：

```text
用户自定义保存目录 > 系统下载目录/LanTransfer
```

规则说明：

- 默认保存目录为系统下载目录下的 `LanTransfer` 文件夹。
- 用户可以在应用内通过系统目录选择器修改默认保存目录。
- 用户修改后的保存目录保存在本地配置中。
- 后续接收文件时，优先保存到用户自定义保存目录。
- 如果用户未设置自定义保存目录，则保存到系统下载目录下的 `LanTransfer` 文件夹。
- v0.1 不支持每次接收文件时临时选择保存目录。

跨平台示例：

```text
Windows: C:\Users\用户名\Downloads\LanTransfer
macOS:   /Users/用户名/Downloads/LanTransfer
Linux:   /home/用户名/Downloads/LanTransfer
```

#### 9.6.2 保存目录可用性检查

应用需要检查保存目录是否可用。

规则说明：

- 应用启动或首次接收文件时检查保存目录。
- 如果保存目录不存在，应用尝试自动创建。
- 如果目录无法创建、无法访问或无写入权限，需要提示用户重新选择保存目录。
- 当保存目录不可用时，设备不允许接收文件。
- v0.1 中，`receiveEnabled` 与保存目录可用性保持一致。

```text
receiveEnabled = 保存目录可用
```

#### 9.6.3 重名与保存失败规则

文件重名时，按 9.3.3 的重名文件处理规则执行：

- 接收端用户选择覆盖后，覆盖全部重名文件。
- 接收端用户选择取消后，取消本次传输任务。
- v0.1 不支持自动重命名。

保存失败时，需要在接收方展示失败原因，并将任务状态标记为失败。

常见失败场景：

- 保存目录不存在且创建失败。
- 保存目录无写入权限。
- 文件写入过程中磁盘空间不足。
- 文件写入过程中应用关闭或传输中断。

### 9.7 传输任务状态

优先级：P0

需求说明：

- 应用展示发送任务和接收任务。
- 每个任务展示文件名、方向、进度、状态。
- 基础状态包括：等待确认、传输中、成功、失败、已拒绝。
- v0.1 不保留历史任务。
- 应用关闭后任务列表清空。

#### 9.7.1 任务展示规则

v0.1 只展示当前应用运行期间的传输任务。

规则说明：

- 任务状态仅保存在内存中。
- 不将任务历史写入本地配置。
- 不展示上一次启动时的历史任务。
- 应用关闭后，任务列表清空。
- 如果应用关闭时存在传输中任务，本次传输中断。
- v0.1 不支持传输恢复。
- v0.1 不支持断点续传。

#### 9.7.2 任务状态

基础任务状态：

| 状态 | 说明 |
| --- | --- |
| 等待确认 | 发送方已提交传输请求，等待接收方确认 |
| 已拒绝 | 接收方拒绝本次传输 |
| 传输中 | 文件正在上传或接收 |
| 成功 | 本次任务全部文件传输完成 |
| 失败 | 本次任务因错误中断或保存失败 |

多文件任务中，任务整体状态以整个任务为单位判断：

- 所有文件上传并保存成功，任务状态为成功。
- 任一文件上传或保存失败，任务状态为失败。
- 接收方拒绝任务，任务状态为已拒绝。

## 12. 初步接口规划

> 本节仅记录方向，后续技术设计阶段细化。

### 12.1 本地服务接口候选

| 接口 | 方法 | 说明 |
| --- | --- | --- |
| `/api/ping` | GET | 测试设备是否在线 |
| `/api/device` | GET | 获取设备名称、版本、服务状态 |
| `/api/transfer/request` | POST | 发送文件传输请求 |
| `/api/transfer/status/:transferId` | GET | 查询传输请求状态 |
| `/api/transfer/upload` | POST | 上传单个文件内容 |

### 12.2 v0.1 技术方案

#### 12.2.1 Rust HTTP 服务

v0.1 使用：

```text
axum + tokio
```

说明：

- `tokio` 作为 Rust 异步运行时，用于网络、异步 I/O、任务调度。
- `axum` 作为本地 HTTP 服务框架，用于实现设备信息、传输请求、状态查询和文件上传接口。
- 该组合属于 Rust 网络生态中的主流方案，适合后续扩展。

#### 12.2.2 前端与 Rust 通信

本机 React 前端与 Rust 之间使用：

```text
Tauri command + Tauri event
```

规则说明：

- React 通过 Tauri command 调用 Rust 本地能力。
- Rust 通过 Tauri event 通知 React 更新状态。
- Zustand 管理前端侧设备信息、连接状态、文件队列和传输任务状态。

#### 12.2.3 发送方上传进度

v0.1 中，前端不直接负责上传文件内容。

推荐流程：

```text
1. React 选择文件。
2. React 将文件路径、目标地址等信息通过 Tauri command 交给 Rust。
3. Rust 读取文件并通过 HTTP 上传。
4. Rust 计算上传进度。
5. Rust 通过 Tauri event 向 React 发送进度事件。
6. React 更新 Zustand 状态并刷新 UI。
```

进度事件示例：

```json
{
  "event": "transfer-progress",
  "transferId": "b9c5c0c6-91df-4a6f-9fd4-74bfb4c9f9c1",
  "fileName": "demo.zip",
  "fileIndex": 1,
  "fileTotal": 3,
  "sentBytes": 10485760,
  "totalBytes": 52428800,
  "percent": 20
}
```

#### 12.2.4 接收方确认通知

接收方 Rust HTTP 服务收到 `/api/transfer/request` 后，不直接由 HTTP 接口完成用户确认，而是通知本机前端。

推荐流程：

```text
1. 发送方向接收方调用 POST /api/transfer/request。
2. 接收方 Rust 创建 pending 任务。
3. 接收方 Rust 生成 transferId。
4. 接收方 Rust 通过 Tauri event 通知 React 展示接收确认。
5. 接收方用户点击接收或拒绝。
6. 接收方 React 通过 Tauri command 更新本机 Rust 中的任务状态。
7. 发送方轮询 GET /api/transfer/status/:transferId 获取状态。
8. 状态为 accepted 后，发送方开始逐个上传文件。
```

#### 12.2.5 WebSocket/SSE 取舍

v0.1 不引入 WebSocket 或 Server-Sent Events。

规则说明：

- 设备间通信使用普通 HTTP 接口。
- 发送方通过轮询 `/api/transfer/status/:transferId` 等待接收方确认。
- 轮询间隔暂定为 1 秒。
- Rust 与本机 React 的实时状态同步通过 Tauri event 完成。

#### 12.2.6 transferId 生成与关联

`transferId` 由接收方生成。

规则说明：

- 发送方调用 `/api/transfer/request` 时不生成最终 `transferId`。
- 接收方收到请求后生成 UUID v4 作为 `transferId`。
- 接收方将 `transferId` 返回给发送方。
- 后续状态查询和文件上传都需要携带该 `transferId`。
- 多文件任务共享同一个 `transferId`。

传输请求响应示例：

```json
{
  "ok": true,
  "transferId": "b9c5c0c6-91df-4a6f-9fd4-74bfb4c9f9c1",
  "status": "pending"
}
```

#### 12.2.7 传输状态轮询

发送方在收到 `transferId` 后，轮询：

```http
GET /api/transfer/status/:transferId
```

状态示例：

```json
{
  "ok": true,
  "transferId": "b9c5c0c6-91df-4a6f-9fd4-74bfb4c9f9c1",
  "status": "accepted"
}
```

状态枚举：

| 状态 | 说明 |
| --- | --- |
| `pending` | 等待接收方确认 |
| `accepted` | 接收方已确认，可以开始上传 |
| `rejected` | 接收方已拒绝 |
| `uploading` | 文件上传中 |
| `completed` | 传输完成 |
| `failed` | 传输失败 |

1. 两台电脑处于同一局域网。
