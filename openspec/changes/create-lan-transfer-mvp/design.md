## Context

本变更定义局域网文件传输工具 v0.1 MVP。产品面向个人多桌面设备场景，目标是在同一局域网内通过手动输入接收方地址完成一个或多个文件的直连传输，减少微信、网盘等第三方中转。

当前仓库已有 PRD 与 UI 参考图，但尚未形成 OpenSpec 规格。v0.1 的实现需要同时覆盖桌面 UI、本地配置、Rust 本地能力、HTTP 接收服务、局域网请求、文件读写和传输状态同步，因此需要先明确跨模块设计约束。

## Goals / Non-Goals

**Goals:**

- 跑通两台桌面设备之间的局域网手动连接和文件传输闭环。
- 使用 Tauri 2 + React + TypeScript + Tailwind CSS 构建单主页面桌面应用。
- 使用 Rust + axum + tokio 提供本地 HTTP 接收服务和文件传输能力。
- 使用 Tauri command/event 连接前端 UI 与 Rust 本地能力。
- 支持本机信息展示、目标连接、最近连接、文件队列、接收确认、保存目录和运行期任务列表。
- 保持 v0.1 实现直接、可调试、可扩展，优先满足开发者本人和个人多设备使用。
- 虽然应用规模较小，但实现应追求“小而美”：代码结构清晰、命名规范、职责边界明确，便于后续学习、调试和迭代。

**Non-Goals:**

- 不支持公网传输、云端中转、账号体系或移动端。
- 不支持自动局域网设备发现、二维码连接、信任设备和传输历史持久化。
- 不支持文件夹传输、断点续传、发送中取消、多文件并发上传或完整端到端加密。
- 不引入 WebSocket、SSE 或复杂配对协议。
- 本变更阶段不生成应用代码，仅定义可实施的规格与任务。

## Decisions

### 使用普通 HTTP 作为设备间传输协议

v0.1 设备间通信使用 HTTP 接口：`GET /api/device`、`POST /api/transfer/request`、`GET /api/transfer/status/:transferId`、`POST /api/transfer/upload`。

选择原因：

- 与手动输入 `IP:端口` 的 MVP 体验匹配。
- 易于调试，可以用浏览器、curl 或日志定位问题。
- axum/tokio 能较自然地处理异步请求和文件流。

备选方案：

- WebSocket/SSE：能减少轮询，但增加连接生命周期和错误处理复杂度，不适合 v0.1。
- 自定义 TCP 协议：性能可控，但调试成本更高，也会增加协议设计负担。

### 发送请求与文件上传分离

发送方先调用 `/api/transfer/request` 提交元数据，接收方确认后，发送方再逐个调用 `/api/transfer/upload` 上传文件内容。

选择原因：

- 接收方可以在上传前查看文件名、数量、大小和发送方信息。
- 接收方可以在真正写入前进行重名检查。
- 拒绝场景无需浪费网络和磁盘 I/O。

备选方案：

- 直接上传文件：流程更短，但接收确认与重名处理不清晰。
- 一次 multipart 上传全部文件：实现更简单但不利于大文件进度和逐文件失败定位。

### transferId 由接收方生成

`transferId` 在接收方收到传输请求后生成 UUID v4，并返回给发送方。后续状态查询和上传都使用该 ID。

选择原因：

- 接收方是任务状态的权威来源。
- 避免发送方生成 ID 与接收方本地任务冲突。
- 便于接收方把 HTTP 请求、Tauri event 和文件写入状态关联起来。

备选方案：

- 发送方生成 ID：减少一次接收方决策，但接收方仍需校验和去重。

### Rust 负责文件上传与保存，React 负责交互呈现

React 通过文件选择或拖拽收集文件路径和用户意图，再通过 Tauri command 交给 Rust。Rust 负责读取文件、发送 HTTP 请求、写入保存目录和计算进度；React 通过 Tauri event 更新 Zustand 状态。

选择原因：

- 避免前端一次性读取大文件进内存。
- 本地文件系统权限、路径处理和流式 I/O 更适合放在 Rust/Tauri 层。
- UI 状态与底层传输状态通过事件解耦。

备选方案：

- 前端直接上传 Blob：Web 模式熟悉，但桌面大文件、本地路径和进度控制受限。

### 本地 HTTP 服务监听 `0.0.0.0:实际端口`

服务端监听所有本机网络接口，UI 展示推荐的局域网 IPv4 地址和实际端口。默认尝试 `7788`，冲突时递增到 `7888`。

选择原因：

- 用户无需理解监听地址和展示地址的差异。
- 多网卡场景下可以通过候选列表切换展示地址，而不影响服务监听。
- 端口自动递增能降低首次启动失败概率。

备选方案：

- 固定监听某个 IP：更严格，但多网卡和网络切换下更容易不可用。
- 固定端口失败即报错：实现简单，但 MVP 可用性较差。

### v0.1 单任务接收、顺序上传多文件

接收方同一时间只允许一个待确认或传输中的任务；多文件任务共享 `transferId`，按待发送列表顺序逐个上传。

选择原因：

- 降低接收确认、重名覆盖、任务状态和文件写入并发复杂度。
- 足以满足个人多设备 MVP 场景。

备选方案：

- 多任务并发接收：体验更强，但需要队列、冲突、磁盘压力和取消策略。
- 多文件并发上传：速度可能更高，但进度和失败恢复更复杂。

### 设备名称编辑规则

v0.1 支持用户编辑自定义设备名称。设备名称最大长度限制为 256 个字符，提交前后需要去除首尾空白；去除空白后为空时不允许保存。

选择原因：

- 256 个字符足以覆盖个人设备名称、备注式命名和少量描述。
- 非空校验可以避免 UI 中出现不可识别设备。
- 该规则简单明确，适合 MVP 阶段实现和测试。

备选方案：

- 更短限制，例如 32 或 64 字符：更利于紧凑 UI，但对备注式命名不够宽松。
- 不限制长度：实现简单，但可能导致布局异常和配置数据质量下降。

### 实现代码质量与注释策略

v0.1 实现时应把可读性和可维护性作为核心约束。Rust 端和前端都需要保持清晰模块边界、稳定类型定义、明确错误处理和一致命名；Rust 端尤其需要补充面向学习和调试的注释。

注释策略：

- 对业务意图、协议约定、状态流转、错误处理和跨线程/异步边界进行说明。
- Rust 代码中对不直观的 API、trait、生命周期、异步任务、stream、channel、Tauri command/event、axum extractor/response 等用法补充学习型注释。
- 注释解释“为什么这样做”和“调用时要注意什么”，避免只复述代码本身。
- 前端代码对复杂状态推导、任务状态映射和 UI 交互边界补充必要说明。

选择原因：

- 该项目既是工具应用，也是 Rust/Tauri 学习项目。
- 更完整的注释能降低后续调试、重构和功能扩展成本。
- 小应用也应保持专业工程质量，避免 MVP 演变成难维护的临时代码。

## Risks / Trade-offs

- 手动输入 IP/端口门槛较高 -> v0.1 通过复制本机地址、最近连接和明确错误提示降低成本，自动发现留到 v0.2。
- HTTP 明文传输安全性有限 -> v0.1 限定局域网个人使用，不处理公网和敏感安全场景，后续再引入信任设备和加密。
- 多网卡推荐可能选错地址 -> 提供候选地址列表和手动切换展示地址，且不改变服务监听范围。
- 接收方单任务限制会拒绝并发请求 -> 用明确忙碌响应提示发送方稍后再试，换取 v0.1 状态模型简单可靠。
- 不支持发送中取消和断点续传 -> 网络中断或应用关闭时任务进入失败，后续版本再扩展取消、临时文件清理和恢复。
- 大文件传输依赖磁盘和网络稳定性 -> 实现阶段必须采用流式读取/写入与进度事件，避免一次性加载完整文件。
- 注释过多可能干扰阅读 -> 注释应集中在 Rust 学习点、跨模块约定和复杂逻辑处，简单赋值、显而易见的 UI 渲染不需要机械注释。

## Migration Plan

当前为新产品能力定义，无既有线上用户或数据迁移。实现时按模块逐步落地：

1. 搭建 Tauri 2 + React + TypeScript + Tailwind CSS 应用骨架。
2. 实现本地配置、设备信息、保存目录和 HTTP 服务启动。
3. 实现主页面 UI 与 Zustand 状态。
4. 实现目标连接测试和最近连接。
5. 实现传输请求、接收确认、状态轮询和顺序上传。
6. 实现保存、重名覆盖、错误反馈和打开保存目录。
7. 使用两台同局域网设备完成端到端验收。

如某个实现阶段出现不可接受的复杂度，应保持规格不扩大，优先缩小技术实现而非加入 v0.2 能力。
